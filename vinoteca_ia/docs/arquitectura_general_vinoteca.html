02

Descripción de cada artefacto

Cada artefacto tiene una tecnología, una responsabilidad única y un contrato claro con el resto del sistema. Ninguno conoce los detalles internos de los demás.

🌐
① API Gateway
FastAPI + Uvicorn · Síncrono

Único punto de entrada al sistema. Autentica cada canal, aplica rate limiting, corre guardrails de entrada (PII, jailbreak), inyecta el Correlation ID y deriva al Orquestador. Expone cuatro endpoints: /chat, /webhook, /aprobar y /health. El streaming de respuestas al cliente también pasa por aquí vía Server-Sent Events.

FastAPIUvicornasynciohttpxSSE streaming
⚙️
② Orquestador — Motor PRAO
Agno Runtime · Stateless · Síncrono

El corazón del sistema. Implementa el bucle Perceive→Reason→Act→Observe usando Agno como framework. Gestiona el circuit breaker (max 5 pasos), las idempotency keys, el Two-Phase Commit, la detección de Stuck State y el Graceful Fallback. Es completamente stateless: todo estado vive en PostgreSQL o Redis, nunca en memoria del proceso.

AgnoPython 3.11+Pydantic v2statelessasync
🔀
③ Agente Enrutador
LLM cognitivo · temp 0.0

Clasificador ultrarrápido. Recibe el mensaje del cliente y determina en un solo paso PRAO a qué agente especializado derivarlo. Confianza mínima 0.85; si no alcanza, emite Acción Nula y pide aclaración. Latencia objetivo: bajo 200ms.

1 paso máximo6 claseshandoff JSON
🍷
④ Agente Sumiller
LLM cognitivo · temp 0.7 / 0.0

Motor de recomendación personalizada. Infiere el perfil del cliente, recupera conocimiento relevante del catálogo vectorial (RAG), verifica stock en SQL antes de recomendar, y selecciona la capa de conocimiento correcta según el perfil. Temperatura 0.7 para el texto de recomendación; 0.0 para cualquier tool call.

ReActRAG + SQL5 capas catálogo3 perfiles
📦
⑤ Agente de Pedidos
LLM cognitivo · temp 0.0 · Two-Phase

Maneja la transacción completa con Two-Phase Commit obligatorio. Fase 1 (preparación): calcula y presenta sin mutar nada. Fase 2 (ejecución): solo después de confirmación explícita del cliente, con idempotency key activa. Todas las mutaciones generan log inmutable.

Two-PhaseIdempotency KeyMercado Pagorollback
📊
⑥ Agente de Inventario
Solo SQL · temp 0.0 · determinista

Consultas exactas de stock, precios y disponibilidad de entrega. Nunca usa RAG vectorial para datos transaccionales. Aplica validación semántica a cada respuesta del SQL para detectar errores encubiertos (precio = 0, campo null, timeout). Formatea la respuesta según el canal de origen.

Solo SQLnunca vectoresvalidación semántica
💬
⑦ Agente de Soporte
LLM cognitivo · temp 0.4

Atiende reclamos, preguntas fuera del flujo estándar y situaciones emocionales. Detecta el tono del cliente antes de responder. Escala a humano si el caso es complejo, emocional o supera 3 reintentos fallidos. El operador recibe el historial completo de la sesión al momento de la escalada.

RAG FAQescalada humanoNPS triggerCRM log
⚖️
⑧ Agente LLM-as-a-Judge
Auditor async · temp 0.0 · nocturno

Auditor completamente asíncrono que corre a las 3:00 AM. Evalúa todas las conversaciones del día con una rúbrica binaria de 6 criterios. Pertenece a una familia de modelos diferente al agente auditado. Los fallos alimentan el Dataset Adversarial y disparan alertas al equipo técnico.

Base de datos relacional · fuente de verdad

Almacena inventario, precios, pedidos, sesiones de agentes (PgAgentStorage), perfiles de cliente, CRM y el log inmutable de transacciones. Es la única fuente de verdad para datos transaccionales. Nunca se consulta por similitud semántica.

Catálogo semántico · RAG

Almacena el catálogo de vinos en formato vectorial con las 5 capas de conocimiento como metadata. Alimentado por el Knowledge Pipeline. Consultado exclusivamente para recomendaciones y maridajes, nunca para precios ni stock.

⑪ Redis
Cache · Estado efímero

Tres usos críticos: (1) Idempotency Keys con TTL de 30 minutos para bloquear dobles ejecuciones. (2) Rate limiting por canal y por usuario. (3) Message Queue (Pub/Sub) para desacoplar el Knowledge Pipeline de los agentes síncronos.

Relaciones y protocolos

Cada comunicación entre artefactos usa el protocolo correcto para su naturaleza: síncrono para respuestas en tiempo real, asíncrono para procesos que no bloquean al usuario, eventos para desacoplar productores de consumidores.

Origen Destino Protocolo Qué viaja Por qué este protocolo

Orquestador Agentes (todos) Agno handoff JSON Mandato firmado + session_id + herramientas inyectadas Zero Trust: el agente receptor no confía en texto libre
Agentes LLM API externa HTTPS streaming Prompt construido + tools schema Streaming permite SSE al cliente sin esperar respuesta completa

Ag. Sumiller Vector Store Embedding similarity Query semántica → top-5 fragmentos con metadata capa Solo texto cualitativo; prohibido para precios o stock
Orquestador Redis Redis SET/GET Idempotency Keys con TTL 30min; rate limit counters Operación O(1) en memoria; TTL automático evita acumulación
Knowledge Pipeline Vector Store Async batch write Fragmentos embebidos + metadata (capa, fuente, fecha) Background; no bloquea el flujo de conversación

Regla de comunicación inter-agente Los agentes no se llaman directamente entre sí. Toda comunicación pasa por el Orquestador, que valida, firma y entrega el mandato. Un agente no puede influenciar a otro fuera del protocolo establecido. Esto es Zero Trust aplicado a la capa cognitiva.

05

Decisiones clave de despliegue

Antes de elegir el proveedor de cloud o el tier de hardware, estas son las cinco decisiones que más impactan el costo operativo y la resiliencia del sistema.

1
¿pgvector o Qdrant para el Vector Store?

Para Tier 1 y 2: pgvector dentro del mismo PostgreSQL. Elimina un servicio extra, reduce costos y la latencia de red. Para Tier 3 con +10.000 vectores y múltiples tenants: Qdrant Cloud dedicado. El código de retrieval no cambia entre ambos.

Tier 1-2: pgvector · Tier 3: Qdrant
2
¿Dónde corre el Knowledge Pipeline?

En Tier 1: cron job en el mismo servidor, fuera del horario pico (2:00–6:00 AM). En Tier 2+: worker separado en segundo plano consumiendo la cola de Redis Pub/Sub. Nunca bloquea el flujo de conversación.

Siempre background · nunca síncrono
3
¿Managed databases o self-hosted?

Para equipos de 1-3 personas: managed siempre (Railway, Neon, Supabase). El tiempo de DBA es más caro que la diferencia de precio. Self-hosted solo si hay un DevOps dedicado y el volumen justifica el ahorro.

Managed hasta Tier 2
4
¿Cómo manejar el costo del LLM API?

Tres palancas de reducción: (1) Caché semántica para preguntas repetidas (frecuente en vinotecas). (2) Modelo pequeño para clasificación y inventory (GPT-4o-mini o Claude Haiku). (3) Temperatura 0.0 en todo lo que no es texto creativo reduce tokens de output. El modelo potente solo para recomendaciones.

Modelo por tarea · no one-size-fits-all
5
¿Cómo garantizar disponibilidad sin Kubernetes?

En Tier 1: proceso supervisado con systemd o supervisord + health check automático. Si el proceso muere, se reinicia solo en segundos. En Tier 2: dos instancias detrás de un load balancer. 99.9% de uptime sin necesidad de K8s.

systemd + health check = suficiente hasta Tier 2
6
¿Qué monitorear desde el día uno?

Cuatro métricas críticas mínimas: (1) /health con ping a DB y LLM. (2) Costo diario del LLM API por desarrollador. (3) Tasa de escaladas a humano (si sube, algo está fallando). (4) Latencia promedio de respuesta (baseline 1.2s, alerta a 8s).

Sin /health, Kubernetes no sabe si la réplica está sana
